F:

Beschreiben Sie die Gründe für die Weiterentwicklung von ADO zu ADO .NET!

A:

ADO -- alte Connected-Welt, COM-Modell, Client-Server Applikationen

ADO .NET -- n-tier Applikationen, neue Disconnected Welt, XML Unterstützung, weltweit verteilte Applikationen, Aufsplittung des zu komplexen Recordset Objekts

F:

Welche Vorteile bieten Mehrschicht Architekturen (three tier, n- tier)?

A:

Bei Drei- oder Mehrschicht Architekturen können die einzelnen Komponenten der Schicht unabhängig voneinander entwickelt werden. Ein Austausch der Anzeige, der Logikkomponenten oder der Datenkomponenten ist problemlos möglich.

F:

Erläutern Sie den Begriff „loosely coupled“!

A:

Durch die lose Kopplung der Komponenten bei Mehrschicht Architekturen landen hintereinander gestellte Clientabfragen nicht unbedingt auf dem gleichen Webserver, Application Server oder DB Server (Lastverteilung).
Durch hinzufügen weiterer Server/ Rechenleistung erhält man eine sehr hohe Skalierbarkeit (service on demand).

F:

Worin unterscheiden sich connected und disconnected Modell in ADO .NET?

A:

Connected DisConnected
Definition Der Nutzer einer Applikation ist ständig mit der einer Datenquelle verbunden. Der Nutzer einer Applikation ist nicht ständig mit einer Datenquelle verbunden. Mobile Nutzer sind die häufigste Zielgruppe. Die Nutzer nehmen einen Teil der Daten mit auf den abgetrennten Computer (Dataset), bearbeiten diese und fügen die geänderten Daten später wieder in die zentrale Datenbank ein.
Vorteil
  • Einfachere Wartung der sicheren Umgebung,
  • gleichzeitiger Zugriff ist einfacher zu kontrollieren (Datenkonsistenz),
  • Daten sind häufiger in einem aktuelleren Zustand als im disconnected Szenario.
  • Arbeit und Verbindung zur Datenbank ist zu jedem beliebigem Zeitpunkt möglich,
  • andere User können ungenutzte Ressourcen (Connections) nutzen,
  • Skalierbarkeit und damit Performaz der Anwendung ist besser (Millionen User möglich).
Nachteil
  • Ständige Netwerkverbindung ist erforderlich (Netzwerk- und Datenbankressourcen),
  • die Skalierbarkeit ist schwierig.
  • Aktualitiät der Daten ist nicht gewährleistet,
  • Datenkonsistenz und konkurrierender Zugriff müssen gelöst werden.

F:

Erklären Sie die Begriffe DataProvider und DataSet aus ADO .NET!

A:

Ein DataProvider ist ein in einer .NET-Sprache geschriebener Datenbanktreiber (Sammlung von .NET Klassen und Interfaces) für einen speziellen Datenbankanbieter (z.B. MS SQL Server, Oracle, MySql etc.).

Ein DataSet ist die logische Abbildung einer Datenbank oder Teilen einer Datenbank im Arbeitsspeicher eines Rechners incl. aller Beziehungen und Constraints. Sie erlaubt das abgekoppelte Arbeiten mit der Datenbank.

F:

Beschreiben Sie den Zugriff auf verschiedene Datenbanksysteme über ADO .NET!

A:

Für die hauseigenen Datenbanksysteme MS-SQL Server und Access stehen mit dem SQL Server .NET Data-Provider bzw. OLE DB .NET Data-Provider eigene Implementierungen von Microsoft zur Verfügung.

Die meisten Datenbankhersteller bieten inzwischen native Implementierung von .NET-Datenbanktreibern an.

Eine weitere Möglichkeit besteht darin, über den ODBC-Treiber auf ältere Datenbanksysteme zurückzugreifen.

F:

Welche Nachteile hat die Verwendung der OLE DB und ODBC .NET Data Provider?

A:

OLE DB bzw. ODBC sind Entwicklungen aus der COM-Zeit des Windows-Betriebssystems und setzen innerhalb von .NET eine oder mehrere weitere Schichten zwischen die Applikation und die Datenbank. Sie erhöhen damit die Fehlermöglichkeit und sind in allgemeinen auch langsamer als native .NET-Datenbank-Provider.

Erklären Sie, wann ein DataReader eingesetzt werden kann!

A:

Ein DataReader kann dann eingesetzt werden, wenn man lediglich einen lesenden Zugriff auf die Daten benötigt. So dient er z.B. zum Füllen von Combo- und Listboxen.

F:

Beschreiben Sie die Aufgaben des Connection Objekts, Command Objekts, Parameter Objekts und des DataReader Objekts

A:

  • Connection

    Ein Connection-Objekt stellt eine eindeutige Sitzung für eine Datenbankserver-Datenquelle dar. Bei einem Client-/Server-Datenbanksystem entspricht dies einer Netzwerkverbindung mit dem Server.

  • Command

    Stellt eine SQL-Anweisung oder eine gespeicherte Prozedur dar, die in einer Datenbank ausgeführt werden soll.

  • Parameter

    Stellt einen Parameter für ein SqlCommand dar. Mit Hilfe der Parameter-Objekte kann man sog. parametrisierte Abfragen erstellen, die WHERE-Statements unnötig machen.

  • DataReader

    Stellt ein Verfahren zum Lesen eines Vorwärtsstreams von Zeilen aus einer Datenbank bereit.

F:

Beschreiben Sie den Zugriff auf die Datensätze über einen DataReader!

A: 
...
// KundenID setzen
this.cmdSelectKundenBuchungen.Parameters["@kd_KundenID"].Value = aKunden[k].KundenID;

// Datenbankabfrage starten
dr = this.cmdSelectKundenBuchungen.ExecuteReader();

// Alle Buchungen des Kunden auslesen
while(dr.Read())
{
	Buchung bu = new Buchung();
	bu.BuchungsID = dr.GetInt32(0);
	bu.GebuchtePlaetze = dr.GetInt32(1);
	bu.Bezahlt = dr.GetBoolean(2);
	bu.Storniert = dr.GetBoolean(3);
	bu.StartTermin = dr.GetDateTime(4);
	bu.StartOrt = dr.GetString(5);
	bu.Preis = dr.GetDecimal(6);
	bu.EventBezeichnung = dr.GetString(7);

	// Buchung beim Kunden einfuegen
	aKunden[k].Buchungen.Add(bu);
	bu = null;
}
// Datareader schliessen
dr.Close()

Obiges Beispiel listet alle Buchungen eines Kunden und speichert diese ab.

Nachdem das SELECT-COMMAND-Objekt zunächst mit dem Parameter für die WHERE-Bedingung versehen wurde, führt die ExecuteReader-Methode des Command-Objektes die Abfrage an das DBMS aus und liefert ein DataReader-Objekt zurück.

Anschließend liefert die Read-Methode des DataReader-Objektes über eine Schleife alle Datensätze aus. Die Spaltenwerte der Datensätze werden im Beispiel an die Attribute eines Buchungsobjektes übergeben.

Zuletzt wird die Verbindung des DataReader-Objektes mit Hilfe der close-Methode geschlossen.

Beschreiben Sie die Zusammenarbeit von DataAdapter, DataSet und DataTables im disconnected Modell!

A:

Der DataAdapter verfügt über ein SELECT-Command-Objekt, welches die Daten aus der Datenbank holt. Die Fill()-Methode des DataAdapters erwartet als Paramter ein DataSet und einen Namen für ein zu erstellendes DataTable-Objekt. Sie erzeugt dann im jeweils übergebenen DataSet eine DataTable.

Beim Update mit der Datenbank werden je nach Änderung das Insert, Update oder Delete Command des DataAdapters aufgerufen, um die Änderungen des Datasets in die Datenbank zu übertragen.

Der DataAdapter stellt daher das Bindeglied zwischen Dataset Tabelle und Datenbank dar. Pro DataTable eines Datasets gibt es in der Regel einen DataAdapter.

F:

Nennen Sie die vier Command - Objekte eines DataAdapters und erläutern Sie deren Aufgabe!

A:

  • SELECT-Command zum Lesen von Datensätzen aus der Datenquelle,

  • UPDATE-Command zum Ändern bestehender Datensätze in der Datenquelle,

  • INSERT-Command zum Einfügen neuer Datensätze in die Datenquelle,

  • DELETE-Command zum Löschen von Datensätzen aus der Datenquelle.

F:

Wie werden Daten in ein DataSet eingelesen?

A:

z.B. über die Fill-Methode eines DataAdapters. Diese liest die Daten aus der Datenbank über das SELECT Command aus.

F:

Auf welche Weise können geänderte Daten eines DataSets in die Datenbank übertragen werden?

A:

Über die Update -Methode des DataAdapters werden automatisch INSERT-, UPDATE- oder DELETE Command für jede Zeile der DataTable aufgerufen. Die Datenbank wird aktualisiert.

F:

Wie viele DataAdapter arbeiten in der Regel mit einem DataSet zusammen?

A:

Pro Tabelle innerhalb eines DataSets steht in der Regel ein DataAdapter zur Verfügung.

F:

Erläutern Sie die Aufgaben des DataSets in einer Clientapplikation!

A:

Das DataSet stellt auf der Client-Seite die Datenstruktur dar, die innerhalb einer entfernten Datensammlung (häufig eine Datenbank) vorzufinden ist.

Das DataSet ist damit die logische Datenbank auf Clientseite, ein Puffer für die Daten.

F:

Welche Collections und Objekte sind in einem DataSet enthalten?

A:

  • Tables

    Das DataSet enthält eine Auflistung all seiner Tabellen.

  • Columns

    Jedes Table-Objekt hat eine Auflistung seiner Spalten.

  • Constraints

    Jedes Table-Objekt verfügt über eine Liste all seiner Einschränkungen, z.B. Foreign Key, Unique Keys, etc.

  • Rows

    Jedes Table-Objekt verfügt über eine Auflistung seiner Datensätze

  • Relations

    Jedes DataSet-Objekt verfügt über eine Auflistung der Beziehungen zwischen seinen Tabellenobjekten.

F:

Erklären Sie den Begriff ORM!

A:

Ein ORM (Object-Relational-Mapping) ermöglicht es, Operationen an einer relationalen Datenbank in einer objektorientierten Programmiersprache auszuführen, ohne dass man sich explizit mit SQL-Anweisungen auseinandersetzen muss.
ORM Tools erstellen für jede Tabelle in der Datenbank eine Klasse und für jede Zelle ein Objekt. Diese Objekte können im Programm genutzt werden, wie jedes andere Objekt, um mit den Daten aus der Datenbank zu interagieren (Mapping).

F:

Erläutern Sie den Zweck des Entity Frameworks!

A:

Das Entity Framework ermöglicht den Datenzugriff aus dem .NET Framework auf relationale Datenbanken.
Datenbankabfragen werden in Form von LINQ Funktionen gechrieben. Diese Methodenaufrufe werden automatisch in SQL Abfragen übersetzt und ausgeführt.

F:

Beschreiben Sie den Unterschied zwischen dem Code First und dem Database First Ansatz!

A:

Bei der Code First Approach wird der Code als Hauptquelle für die Definition der Datenbankstruktur verwendet. Dies bedeutet, dass der Entwickler Klassen in der Anwendung schreibt, die das Datenmodell darstellen und Entity Framework verwendet diese Klassen, um die Datenbankstruktur zu erstellen.

  • Vorteil: vollständige Kontrolle über die Struktur der Datenbank.
  • Nachteil: Zeitaufwand für die manuelle Definition der Datenbankstruktur.

    • Bei der Database First Approach wird die vorhandene Datenbank als Hauptquelle für die Definition des Datenmodells verwendet. Das Entity Framework analysiert die Struktur der vorhandenen Datenbank und generiert die Klassen in der Anwendung , die das Datenmodell darstellen.

  • Vorteil: der Entwickler kann schnell mit der Entwicklung von Anwendungsfunktionen beginnen, ohne sich Gedanken über die manuelle Definition der Datenbankstruktur machen zu müssen.
  • Nachteil: weniger Kontrolle über die Struktur des Datenmodells, Schwierigkeiten bei Änderungen an der Datenbankstruktur.

    Welchen Zweck erfüllen Data Annotations?

    A:

    Data Annotations sind .NET Attribute, die verwendet werden, um die Struktur und das Verhalten von Klassen und Eigenschaften im Code zu definieren. Häufig verwendete Annotatins sind z.B. Key, Foreign-Key(…) oder Required.

    Die DbContext Klasse dient als Verbindungspunkt zwischen Anwendung und Datenbank.
